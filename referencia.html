¿Qué evalúa el dashboard?

Mide el cumplimiento documental de cada directorio (proyecto o requerimiento) en la biblioteca Demanda y Portafolio TI. Por cada carpeta del repositorio existe un catálogo de artefactos obligatorios, y se calcula qué porcentaje están efectivamente presentes como archivos.

El dashboard se organiza en tres niveles según la audiencia:

  • 📊 Ejecutivo — vista agregada del portafolio para gerencia y dirección.
  • 🗂 Explorador — navegación por Negocio → Año → Directorio para coordinadores.
  • 🔍 Ver por código — ficha individual con artefactos faltantes para jefes de proyecto.

Dos formas de medir completitud — ¿cuál mirar?

🎯 % Completitud (según fase)

Es la métrica recomendada para evaluar a un directorio. Se ajusta al tipo de iniciativa detectado: solo considera los artefactos de la fase activa, no castiga a un directorio joven por no tener documentos de fases que aún no le tocan.

Proyecto → mide contra Demanda + Proyecto Requerimiento → mide contra Demanda + Requerimiento Solo Demanda → mide solo contra Demanda

📚 % Completitud (catálogo total)

Denominador fijo contra los 46 artefactos obligatorios del catálogo completo. Útil como referencia absoluta, pero penaliza a directorios que aún están en fases tempranas.

Reglas de conteo de archivos

✅ Sí cuentan para la completitud

  • Archivos que están a nivel de la carpeta esperada (ej. Proyecto/1_Gestion/Plan_*.docx).
  • Archivos cifrados con FRP — el sistema descifra el nombre lógico quitando la extensión .frpenc antes de comparar.
  • Archivos cuyo nombre coincide con la nomenclatura esperada según el catálogo.
  • Archivos genéricos (con cualquier nombre) cuando el catálogo dice "No definido" para ese artefacto.

❌ NO cuentan para la completitud

  • Archivos dentro de la subcarpeta historico/ o histórico/ — son versiones anteriores archivadas.
  • Archivos de sistema: readme.txt, thumbs.db, desktop.ini, .DS_Store.
  • Artefactos que viven fuera del repositorio (Project Online, enlaces externos a otros sistemas) — no son medibles desde aquí.

Por qué la carpeta histórico no cuenta

La convención del repositorio es que la versión vigente del documento debe estar visible a nivel de la carpeta correspondiente. Las versiones anteriores se mueven a la subcarpeta historico/ para no confundir cuál es la actual.

Por eso solo cuenta como presente el archivo que está a nivel de la carpeta; las copias dentro de historico/ no se cuentan, aunque sean documentos válidos.

/Proyecto/1_Gestion/ ├── Plan_Gestion_Proyecto_Inf_ABC.docx ✅ Cuenta (vigente) ├── Acta_proyecto_ABC.docx ✅ Cuenta (vigente) └── historico/ ├── Plan_Gestion_v1.docx ❌ No cuenta (archivado) ├── Plan_Gestion_v2.docx ❌ No cuenta (archivado) └── 2024/ └── Acta_old.docx ❌ No cuenta (archivado)

Cómo el sistema reconoce un artefacto

Para detectar que un archivo presente equivale a un artefacto del catálogo, el sistema usa un método de doble verificación:

  • Si el catálogo define una nomenclatura específica (ej. FN_[Cód.Nec], Acta_proyecto_[Código Proyecto].doc) → el sistema convierte el patrón en una expresión regular y busca archivos cuyo nombre coincida exactamente.
  • Si el catálogo dice "No definido" para la nomenclatura → el sistema cuenta por cantidad: si se esperan N artefactos y hay al menos N archivos genéricos en la carpeta, se asume que están presentes.
⚠️ Implicación importante: si el nombre de un archivo no respeta la nomenclatura esperada, el sistema NO lo reconoce como el artefacto correspondiente, aunque el contenido sea correcto. Por eso es crítico respetar la convención de nombres al subir documentos al repositorio. Esto se ve reflejado en la ficha de cada directorio como un artefacto faltante.

Hitos con artefactos alternativos (OR-sets)

Algunas carpetas del catálogo permiten cumplir un hito con cualquiera de varios formatos alternativos. En estos casos, basta con tener al menos uno de los formatos para que el hito se considere cumplido — no se exigen todos.

El sistema reconoce los archivos por su prefijo y los asigna automáticamente al hito correcto.

📍 Demanda / 2_Entend_o_Consenso

Esta carpeta tiene 2 hitos, cada uno cumplible con cualquiera de 3 formatos:

Hito 1: Entendimiento inicial

Basta con uno de:

  • Archivo MPN_*.docx (Modelo de Procesos de Negocio)
  • Archivo DEF_*.docx (Definición funcional)
  • Archivo HU_*.xlsx (Historias de usuario)

El nombre NO debe contener "consensu" para clasificar como inicial.

Hito 2: Entendimiento consensuado

Basta con uno de:

  • Archivo MPN_*_consensuado.*
  • Archivo DEF_*_consensuado.*
  • Archivo HU_*_consensuado.*

El nombre SÍ debe contener "consensu" para clasificar como consensuado.

📍 Proyecto / 2_Producto / 2_Requerimiento (también aplica a Requerimiento / 2_Producto / 2_Requerimiento)

El catálogo define un único artefacto llamado "RIN/ Historia de usuario (HU)", que se cumple con cualquiera de los dos formatos:

  • Archivo RIN_*.docx (Especificación de requerimiento)
  • Archivo HU_*.xlsx (Historia de usuario)
💡 ¿Cómo se ve esto en la ficha? Cuando un hito OR-set está cumplido, aparece en "Artefactos detectados" con un badge azul ("cumplido vía MPN") y el nombre del archivo que cumplió. Cuando no está cumplido, aparece en "Artefactos faltantes" con un badge amarillo ("hito alternativo") y la lista de formatos aceptados.

Subcarpetas no oficiales — el sistema sí las reconoce

Si organizas tus archivos dentro de subcarpetas adicionales que no están definidas en el catálogo (por ejemplo 1_Gestion/Versiones/, 2_Producto/Borrador/, o 1_Gestion/2024/), el sistema los detecta y cuenta para la completitud mediante búsqueda recursiva.

Sin embargo, te recomendamos mantener los archivos a nivel de las carpetas oficiales del catálogo. Cuando un archivo está en una subcarpeta no oficial:

  • Aparece como presente en la ficha del directorio (cuenta para la completitud).
  • Se marca con un aviso "⚠️ Encontrado en subcarpeta no oficial" en "Artefactos detectados".
  • Se incluye en una alerta global al inicio de la ficha que sugiere reorganizar los archivos.
/Proyecto/1_Gestion/ ├── Plan_Gestion.docx ✅ Cuenta — ubicación oficial ├── Versiones/ │ └── Plan_Gestion_v2.docx ✅ Cuenta — pero alerta de ubicación no oficial ├── 2024/ │ └── Acta.docx ✅ Cuenta — pero alerta de ubicación no oficial └── historico/ └── Plan_Gestion_v1.docx ❌ No cuenta — versión archivada
💡 Casos especiales: las subcarpetas que SÍ están en el catálogo (como 2_Producto/MPN, 2_Producto/DEF, 2_Requerimiento) son ubicaciones oficiales y no generan alerta.

🔬 Cómo el sistema decide si un archivo cumple un artefacto

Esta sección explica con detalle cómo el sistema relaciona los archivos que tú subes al repositorio con los artefactos del catálogo. La idea es que cualquier resultado del dashboard sea reproducible y comprensible: si un archivo aparece como presente o faltante, debe quedar claro por qué.

Las tres capas de detección, en orden

Para cada artefacto esperado del catálogo, el sistema busca un archivo correspondiente aplicando estas tres capas en secuencia. Si la primera encuentra match, no usa las otras. Si falla, intenta con la siguiente.

  1. Capa 1 — Nomenclatura exacta (regex)
    El sistema verifica si el nombre del archivo cumple el patrón oficial del catálogo (por ejemplo Plan_Gestion_Proyecto_Inf_[Código].docx). Si encaja, el artefacto se cuenta como presente sin alertas adicionales. Es el match más confiable.
  2. Capa 2 — Heurística por palabras del nombre del artefacto
    Si el archivo no respeta la nomenclatura, el sistema intenta detectarlo comparando las palabras significativas del archivo con las palabras del nombre del artefacto en el catálogo. Si la coincidencia supera un umbral, el archivo se cuenta como presente, pero con una alerta visible (badge naranja "📝 nombre no estándar") y una sugerencia para renombrarlo. Esta capa se explica en detalle más abajo.
  3. Capa 3 — Genérico (cualquier archivo en la carpeta)
    Solo aplica cuando el catálogo dice "Nomenclatura: No definido". En esos casos, cualquier archivo en la carpeta correcta cuenta como presente con la nota "sin nomenclatura específica". Es la capa más permisiva.

Capa 2 al detalle: cómo funciona la heurística

La heurística es la capa más importante porque recupera muchos archivos que en la práctica están bien subidos pero con nombres no estándar. Para mantener la transparencia, aquí está exactamente qué hace el sistema:

Paso 1 — Normalizar ambos textos

Tanto el nombre del archivo como el nombre del artefacto se normalizan así:

  • Se eliminan las tildes (Gestión → gestion).
  • Se convierte todo a minúsculas (ACTA → acta).
  • Los separadores como guión bajo, guión, paréntesis, corchetes y puntos se tratan como espacios (Plan_Gestion-Proyecto → plan gestion proyecto).
  • Se quita la extensión del archivo (.docx, .pdf, .frpenc, etc.).

Paso 2 — Eliminar palabras vacías (stop words)

Las siguientes palabras no aportan significado y se ignoran en ambos textos:

de, del, la, las, el, los, en, para, con, por, y, o, a, es, ser, una, uno, su, sus, al, un, este, esta, sobre, que, si, no, etc

También se eliminan las secuencias puramente numéricas (códigos de proyecto, fechas, versiones).

Paso 3 — Expandir abreviaciones SUNAT

Si el nombre del archivo o del artefacto contiene una abreviación conocida, el sistema también considera sus palabras expandidas. Por ejemplo, un archivo llamado FNN_001.docx se trata como si dijera "ficha necesidad", y por eso puede matchear con el artefacto "Ficha de necesidad". Esta es la lista completa:

Paso 4 — Calcular un puntaje de similitud (score)

El sistema compara las palabras del archivo con las del artefacto y calcula un puntaje de coincidencia. No todas las palabras valen lo mismo:

  • Peso 1.0 (palabras discriminadoras): palabras que aparecen en pocos artefactos del catálogo. Son las que realmente identifican un artefacto único. Ejemplos: "constitución", "interesados", "lecciones", "memorándum", "cronograma".
  • Peso 0.6 (algo genéricas): palabras que aparecen en 3 o 4 artefactos del catálogo. Aportan algo, pero no son únicas. Ejemplos: "necesidad", "definición", "consensuado", "inicial".
  • Peso 0.3 (genéricas): palabras que aparecen en 5 o más artefactos. No discriminan bien. Ejemplo: "modelo".

El puntaje se calcula así:

score = (suma de pesos de palabras coincidentes) / (suma de pesos de palabras del artefacto)

El score va de 0.00 (ninguna palabra coincide) a 1.00 (todas las palabras del artefacto están en el archivo).

Paso 5 — Aplicar dos reglas de validación

Para que la coincidencia se acepte como match real, debe cumplir ambas condiciones:

  1. Score ≥ 0.65: al menos el 65% del peso de las palabras del artefacto debe estar en el archivo.
  2. Al menos una palabra discriminadora coincide (peso 1.0): esto evita falsos positivos donde solo coinciden palabras genéricas. Por ejemplo, un archivo "Acta de reunión" no debería contar como "Plan de Gestión del Proyecto" solo porque ambos comparten "proyecto" o "gestión".

Paso 6 — Evitar la doble asignación

Un mismo archivo no puede satisfacer dos artefactos distintos. Si el sistema ya asignó un archivo a un artefacto, ese archivo queda fuera de candidatos para el resto. Esto previene que un solo archivo "infle" la completitud del directorio cumpliendo varios artefactos a la vez.

Ejemplo paso a paso (caso real del repositorio)

Supongamos que el catálogo espera el artefacto "Plan de Gestión del Proyecto" en la carpeta Proyecto/1_Gestion. En el directorio del JPI hay un archivo llamado:

Plan de Gestión de Proyectos Informáticos - INSI2025001.docx

El sistema lo procesa así:

  1. Capa 1 (regex): el archivo no respeta el patrón Plan_Gestion_Proyecto_Inf_[Código]. Falla.
  2. Capa 2 (heurística):
    • Palabras del archivo (normalizadas, sin stop words): {plan, gestion, proyectos, informaticos, insi2025001}
    • Palabras del artefacto (normalizadas, sin stop words): {plan, gestion, proyecto}
    • Palabras coincidentes: {plan, gestion} (la palabra "proyecto" no coincide con "proyectos" exactamente; el sistema usa coincidencia exacta de tokens).
    • Pesos: plan=0.6, gestion=1.0, proyecto=0.6. Suma del artefacto = 2.2. Suma de coincidentes = 1.6. Score = 1.6/2.2 = 0.73.
    • ¿Pasa el umbral 0.65? Sí. ¿Tiene al menos una palabra discriminadora (peso 1.0)? Sí ("gestion"). Match aceptado.

El artefacto se cuenta como presente, pero en la ficha aparece el badge naranja con la sugerencia de renombrar el archivo a la nomenclatura oficial.

Limitaciones que debes conocer

La heurística no es perfecta y conviene ser explícitos sobre sus límites:

  • Coincidencia exacta de tokens: el sistema considera "proyecto" y "proyectos" como palabras distintas. No hace análisis morfológico avanzado.
  • Falsos positivos posibles: si dos artefactos del catálogo comparten varias palabras significativas, un archivo puede ser asignado al "incorrecto" desde tu perspectiva. Si detectas un caso así, repórtalo para ajustar la configuración.
  • Falsos negativos posibles: si el JPI usó un nombre completamente distinto (ejemplo: "documento1.docx") sin ninguna palabra del nombre del artefacto, el sistema no podrá identificarlo aunque el contenido sea correcto. En esos casos solo la inspección manual lo detectaría.
  • El sistema no abre archivos: solo evalúa nombres. No revisa el contenido ni los metadatos del documento.
  • Las abreviaciones reconocidas son una lista cerrada: si tu equipo usa otra abreviación que no está en la tabla de arriba, no será reconocida hasta que se agregue a la configuración del sistema.
💡 Recomendación para JPI: aunque el sistema reconozca tu archivo por similitud, sigue la nomenclatura oficial cuando sea posible. Esto facilita la búsqueda manual, evita ambigüedades cuando hay archivos con nombres parecidos y mantiene la documentación ordenada.
🔍 ¿Crees que un match es incorrecto? Si en la ficha de tu directorio aparece un artefacto detectado por similitud que no corresponde, repórtalo al equipo responsable del dashboard indicando: el código del directorio, el nombre del artefacto, y el nombre del archivo en cuestión. Con esa información se puede ajustar el comportamiento.

📘 Términos del dashboard

Directorio
Carpeta con código de proyecto o requerimiento (ej. SNATI2026001, REQ2026005). Es la unidad básica de evaluación del dashboard.
Iniciativa
Sinónimo de directorio cuando se quiere referir a su contenido funcional. Puede ser un Proyecto, un Requerimiento, o estar Solo en Demanda.
Negocio
Área o dominio funcional de TI (ej. SNATI, SNAP). Es el primer segmento de la ruta del directorio en el repositorio.
Fase
Etapa del ciclo de vida documental: Demanda, Proyecto o Requerimiento. Determina qué artefactos del catálogo aplican al directorio.
Tipo de iniciativa
Se deduce automáticamente según las carpetas presentes en el directorio: "Proyecto" si tiene carpeta Proyecto/, "Requerimiento" si tiene Requerimiento/, "Solo_Demanda" si solo hay fase Demanda.
Artefacto
Documento esperado según el catálogo (ej. Plan de Gestión, Acta de Constitución). Cada artefacto tiene una nomenclatura esperada y un responsable.
Nomenclatura
Patrón de nombre que el archivo debe respetar para ser reconocido como un artefacto específico (ej. FN_[Cód.Nec], Acta_proyecto_[Código Proyecto].doc).
Completitud
Porcentaje de artefactos obligatorios presentes sobre los esperados. Es la métrica principal del dashboard.
Promedio ponderado
Para los % agregados (por Negocio, por Año, global) se suman los presentes y los esperados de todos los directorios hijos y se divide. Evita el sesgo del "promedio de promedios" que daría peso igual a directorios pequeños y grandes.
Directorios similares (pares)
Otros directorios del mismo Negocio y mismo Año. Se usan como referencia comparativa en la ficha individual.
Archivos vigentes
Documentos a nivel de la carpeta esperada (cuentan para la completitud).
Archivos archivados
Documentos dentro de la subcarpeta historico/ (no cuentan, son versiones anteriores).
