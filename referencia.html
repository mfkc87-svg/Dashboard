¿Qué evalúa el dashboard?

Mide el cumplimiento documental de cada directorio (proyecto o requerimiento) en la biblioteca Demanda y Portafolio TI. Por cada carpeta del repositorio existe un catálogo de artefactos obligatorios, y se calcula qué porcentaje están efectivamente presentes como archivos.

El dashboard se organiza en tres niveles según la audiencia:

  • 📊 Ejecutivo — vista agregada del portafolio para gerencia y dirección.
  • 🗂 Explorador — navegación por Negocio → Año → Directorio para coordinadores.
  • 🔍 Ver por código — ficha individual con artefactos faltantes para jefes de proyecto.

Dos formas de medir completitud — ¿cuál mirar?

🎯 % Completitud (según fase)

Es la métrica recomendada para evaluar a un directorio. Se ajusta al tipo de iniciativa detectado: solo considera los artefactos de la fase activa, no castiga a un directorio joven por no tener documentos de fases que aún no le tocan.

Proyecto → mide contra Demanda + Proyecto Requerimiento → mide contra Demanda + Requerimiento Solo Demanda → mide solo contra Demanda

📚 % Completitud (catálogo total)

Denominador fijo contra los 46 artefactos obligatorios del catálogo completo. Útil como referencia absoluta, pero penaliza a directorios que aún están en fases tempranas.

Reglas de conteo de archivos

✅ Sí cuentan para la completitud

  • Archivos que están a nivel de la carpeta esperada (ej. Proyecto/1_Gestion/Plan_*.docx).
  • Archivos cifrados con FRP — el sistema descifra el nombre lógico quitando la extensión .frpenc antes de comparar.
  • Archivos cuyo nombre coincide con la nomenclatura esperada según el catálogo.
  • Archivos genéricos (con cualquier nombre) cuando el catálogo dice "No definido" para ese artefacto.

❌ NO cuentan para la completitud

  • Archivos dentro de la subcarpeta historico/ o histórico/ — son versiones anteriores archivadas.
  • Archivos de sistema: readme.txt, thumbs.db, desktop.ini, .DS_Store.
  • Artefactos que viven fuera del repositorio (Project Online, enlaces externos a otros sistemas) — no son medibles desde aquí.

Por qué la carpeta histórico no cuenta

La convención del repositorio es que la versión vigente del documento debe estar visible a nivel de la carpeta correspondiente. Las versiones anteriores se mueven a la subcarpeta historico/ para no confundir cuál es la actual.

Por eso solo cuenta como presente el archivo que está a nivel de la carpeta; las copias dentro de historico/ no se cuentan, aunque sean documentos válidos.

/Proyecto/1_Gestion/ ├── Plan_Gestion_Proyecto_Inf_ABC.docx ✅ Cuenta (vigente) ├── Acta_proyecto_ABC.docx ✅ Cuenta (vigente) └── historico/ ├── Plan_Gestion_v1.docx ❌ No cuenta (archivado) ├── Plan_Gestion_v2.docx ❌ No cuenta (archivado) └── 2024/ └── Acta_old.docx ❌ No cuenta (archivado)

Cómo el sistema reconoce un artefacto

Para detectar que un archivo presente equivale a un artefacto del catálogo, el sistema usa un método de doble verificación:

  • Si el catálogo define una nomenclatura específica (ej. FN_[Cód.Nec], Acta_proyecto_[Código Proyecto].doc) → el sistema convierte el patrón en una expresión regular y busca archivos cuyo nombre coincida exactamente.
  • Si el catálogo dice "No definido" para la nomenclatura → el sistema cuenta por cantidad: si se esperan N artefactos y hay al menos N archivos genéricos en la carpeta, se asume que están presentes.
⚠️ Implicación importante: si el nombre de un archivo no respeta la nomenclatura esperada, el sistema NO lo reconoce como el artefacto correspondiente, aunque el contenido sea correcto. Por eso es crítico respetar la convención de nombres al subir documentos al repositorio. Esto se ve reflejado en la ficha de cada directorio como un artefacto faltante.

Hitos con artefactos alternativos (OR-sets)

Algunas carpetas del catálogo permiten cumplir un hito con cualquiera de varios formatos alternativos. En estos casos, basta con tener al menos uno de los formatos para que el hito se considere cumplido — no se exigen todos.

El sistema reconoce los archivos por su prefijo y los asigna automáticamente al hito correcto.

📍 Demanda / 2_Entend_o_Consenso

Esta carpeta tiene 2 hitos, cada uno cumplible con cualquiera de 3 formatos:

Hito 1: Entendimiento inicial

Basta con uno de:

  • Archivo MPN_*.docx (Modelo de Procesos de Negocio)
  • Archivo DEF_*.docx (Definición funcional)
  • Archivo HU_*.xlsx (Historias de usuario)

El nombre NO debe contener "consensu" para clasificar como inicial.

Hito 2: Entendimiento consensuado

Basta con uno de:

  • Archivo MPN_*_consensuado.*
  • Archivo DEF_*_consensuado.*
  • Archivo HU_*_consensuado.*

El nombre SÍ debe contener "consensu" para clasificar como consensuado.

📍 Proyecto / 2_Producto / 2_Requerimiento (también aplica a Requerimiento / 2_Producto / 2_Requerimiento)

El catálogo define un único artefacto llamado "RIN/ Historia de usuario (HU)", que se cumple con cualquiera de los dos formatos:

  • Archivo RIN_*.docx (Especificación de requerimiento)
  • Archivo HU_*.xlsx (Historia de usuario)
💡 ¿Cómo se ve esto en la ficha? Cuando un hito OR-set está cumplido, aparece en "Artefactos detectados" con un badge azul ("cumplido vía MPN") y el nombre del archivo que cumplió. Cuando no está cumplido, aparece en "Artefactos faltantes" con un badge amarillo ("hito alternativo") y la lista de formatos aceptados.

📘 Términos del dashboard

Directorio
Carpeta con código de proyecto o requerimiento (ej. SNATI2026001, REQ2026005). Es la unidad básica de evaluación del dashboard.
Iniciativa
Sinónimo de directorio cuando se quiere referir a su contenido funcional. Puede ser un Proyecto, un Requerimiento, o estar Solo en Demanda.
Negocio
Área o dominio funcional de TI (ej. SNATI, SNAP). Es el primer segmento de la ruta del directorio en el repositorio.
Fase
Etapa del ciclo de vida documental: Demanda, Proyecto o Requerimiento. Determina qué artefactos del catálogo aplican al directorio.
Tipo de iniciativa
Se deduce automáticamente según las carpetas presentes en el directorio: "Proyecto" si tiene carpeta Proyecto/, "Requerimiento" si tiene Requerimiento/, "Solo_Demanda" si solo hay fase Demanda.
Artefacto
Documento esperado según el catálogo (ej. Plan de Gestión, Acta de Constitución). Cada artefacto tiene una nomenclatura esperada y un responsable.
Nomenclatura
Patrón de nombre que el archivo debe respetar para ser reconocido como un artefacto específico (ej. FN_[Cód.Nec], Acta_proyecto_[Código Proyecto].doc).
Completitud
Porcentaje de artefactos obligatorios presentes sobre los esperados. Es la métrica principal del dashboard.
Promedio ponderado
Para los % agregados (por Negocio, por Año, global) se suman los presentes y los esperados de todos los directorios hijos y se divide. Evita el sesgo del "promedio de promedios" que daría peso igual a directorios pequeños y grandes.
Directorios similares (pares)
Otros directorios del mismo Negocio y mismo Año. Se usan como referencia comparativa en la ficha individual.
Archivos vigentes
Documentos a nivel de la carpeta esperada (cuentan para la completitud).
Archivos archivados
Documentos dentro de la subcarpeta historico/ (no cuentan, son versiones anteriores).
